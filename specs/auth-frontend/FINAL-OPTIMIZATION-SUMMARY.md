## 🎯 应用的核心原则

### 1. 单一事实来源（Single Source of Truth）

**实施方式**：

```
之前：
  - TanStack Query 在 3 个文件中重复
  - 表单处理在 4 个文件中重复
  - 测试模式在 3 个文件中重复

之后：
  design.md (决策) + examples/api-hooks.md (完整示例)
  design.md (决策) + examples/form.md (完整示例)
  testing.md (策略) + examples/component-test.md (完整示例)
```

**效果**：

- ✅ 每个概念只在一处详细说明
- ✅ 更新时只需修改一处
- ✅ 避免文档不一致

### 2. 抽象高度校准

**移除的内容**：

- ❌ React 基础用法（useState、useEffect）
- ❌ TypeScript 基础类型
- ❌ 基础的文件操作步骤
- ❌ 标准的错误处理（try-catch）

**保留的内容**：

- ✅ 团队约定和决策
- ✅ 技术选型和理由
- ✅ 关键模式和最佳实践
- ✅ 常见问题和解决方案

**效果**：

- ✅ 资深开发者可直接使用
- ✅ 新手可参考完整示例
- ✅ 减少维护成本

### 3. 示例优于规则

**优化方式**：

```
之前：
  - 3 段文字描述表单验证
  - 2 个代码片段展示 API 用法
  - 4 个测试模式说明

之后：
  - 1 个完整表单示例（可运行）
  - 1 个完整 API Hook 示例（可运行）
  - 1 个完整测试示例（可运行）
```

**示例特点**：

- ✅ 包含所有必要的导入
- ✅ 完整的类型定义
- ✅ 错误处理展示
- ✅ 最佳实践和反模式对比
- ✅ 可直接复制使用

### 4. 开发者上下文分类

**组织方式**：按 Jiang & Nam (2025) 五分类法

| 分类         | 文件位置                  | 内容                           |
| :----------- | :------------------------ | :----------------------------- |
| **约定**     | AGENTS.md                 | 代码风格、命名规范、文件组织   |
| **指南**     | workflow.md、decisions.md | 开发流程、决策标准、测试策略   |
| **项目信息** | design.md、README.md      | 技术栈、架构、依赖、领域知识   |
| **LLM 指令** | AGENTS.md、prompts.md     | 行为约束、输出格式、决策优先级 |
| **示例**     | examples/\*.md            | 完整可运行示例                 |

**导航工具**：

- ✅ CONTEXT-MAP.md - 快速定位
- ✅ 场景化查找
- ✅ 快速参考卡片

## 🎯 使用新模版的建议流程

### 场景 1：开始新功能开发

```bash
1. 阅读 README.md（5 分钟）
   → 了解模版结构和使用方式

2. 查看 CONTEXT-MAP.md（5 分钟）
   → 快速定位所需信息

3. 阅读 design.md（15 分钟）
   → 理解用户故事、UI/UX、技术设计

4. 参考 examples/（按需）
   → 查看相关完整示例

5. 开始开发
```

**总时间**：约 25-30 分钟（vs 之前 2+ 小时）

### 场景 2：实现 API 集成

```bash
1. 查看 design.md "API 集成" 章节
   → 理解关键模式和决策点

2. 阅读 examples/api-hooks.md
   → 复制完整示例

3. 根据需求修改
   → 快速实现

4. 使用 AUDIT-CHECKLIST.md 检查
   → 确保质量
```

**总时间**：约 15-20 分钟（vs 之前 1+ 小时）

### 场景 3：编写组件测试

```bash
1. 查看 testing.md 测试策略
   → 理解测试金字塔

2. 阅读 examples/component-test.md
   → 复制完整示例

3. 修改为适配组件
   → 快速完成

4. 运行测试验证
   → 确保通过
```

**总时间**：约 10-15 分钟（vs 之前 30+ 分钟）
