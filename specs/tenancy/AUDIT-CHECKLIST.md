# 后端模版审计清单

基于上下文工程最佳实践（Jiang & Nam, 2025; Anthropic, 2025），用于审查后端规格文档质量。

---

## 一、单一事实来源审查

### 检查重复内容

| 问题                      | 若是，则                      | 检查文件                  |
| :------------------------ | :---------------------------- | :------------------------ |
| 同一 DDD 模式在多处解释？ | 保留最权威位置，其他处引用    | design.md ↔ workflow.md  |
| 相同测试模式出现多次？    | 保留在 testing.md，其他处引用 | testing.md ↔ workflow.md |
| CQRS 模式重复说明？       | 统一在 design.md，其他处引用  | design.md ↔ prompts.md   |
| Handler 实现重复？        | 引用 examples/                | workflow.md ↔ examples/  |

### 具体检查项

- [ ] **领域层模式**：design.md、workflow.md、testing.md 中是否重复？
- [ ] **应用层模式**：design.md、workflow.md、prompts.md 中是否重复？
- [ ] **测试模式**：testing.md、workflow.md 中是否重复？
- [ ] **Handler 模式**：workflow.md、prompts.md 中是否重复？

## 二、抽象高度审查

### 校准测试

> **核心问题**：这个细节程度，资深后端开发者会需要，还是能自行推断？

| 内容类型     | 判断标准             | 行动                |
| :----------- | :------------------- | :------------------ |
| **显而易见** | 资深开发者不需要说明 | 删除                |
| **需要提醒** | 资深开发者可能忽略   | 保留简短提示        |
| **需要解释** | 资深开发者需要上下文 | 保留完整说明        |
| **需要示例** | 复杂模式或团队约定   | 保留示例 + 简短说明 |

### 具体检查项

#### 应该删除的内容（显而易见）

- [ ] NestJS 基础概念（如 Controller、Service 用法）
- [ ] TypeScript 基础类型定义
- [ ] 基本的文件操作步骤
- [ ] 标准的错误处理（如 try-catch）
- [ ] 显而易见的 Import 语句

#### 应该精简的内容（需要提醒）

- [ ] 聚合根创建模式（保留关键决策点）
- [ ] 领域事件触发模式（保留策略而非详细步骤）
- [ ] Repository 实现模式（保留约定而非解释）
- [ ] 测试 AAA 模式（保留模式名称而非详细说明）

#### 应该保留的内容（需要解释）

- [ ] DDD 战术设计决策
- [ ] CQRS 模式应用场景
- [ ] 业务规则和约束
- [ ] 领域事件设计

#### 应该强化的内容（需要示例）

- [ ] 完整的聚合根实现示例
- [ ] 端到端的 Command 流程示例
- [ ] 复杂业务规则测试示例
- [ ] 领域事件处理示例

## 三、示例优于规则审查

### 检查规则是否可以转为示例

| 规则类型     | 是否可以转为示例 | 建议                  |
| :----------- | :--------------- | :-------------------- |
| DDD 模式规则 | ✅ 可以          | 用完整聚合根示例展示  |
| 测试模式     | ✅ 可以          | 用实际测试代码示例    |
| Handler 结构 | ✅ 可以          | 用完整 Handler 示例   |
| 业务规则     | ⚠️ 部分可以      | 用场景示例 + 简短规则 |

### 具体检查项

- [ ] 是否有 3 段以上文字解释一个 DDD 概念？→ 改为示例
- [ ] 是否有多个相似规则列表？→ 合并为一个示例
- [ ] 是否有复杂流程用文字描述？→ 改为流程图 + 示例
- [ ] 是否有模式用抽象规则定义？→ 改为代码示例

## 四、防御性重复审查

### 识别过度强调

| 强调方式                 | 问题       | 建议             |
| :----------------------- | :--------- | :--------------- |
| `MUST` / `NEVER`         | 防御性重复 | 删除，说一次即可 |
| `IMPORTANT` / `CRITICAL` | 过度强调   | 删除，信任模型   |
| 多次重复同一规则         | 不信任模型 | 保留一次即可     |

### 具体检查项

- [ ] 检查所有 `MUST`、`NEVER`、`ALWAYS` 关键词
- [ ] 检查是否有重复 3 次以上的规则
- [ ] 检查是否有加粗强调但已说明的内容

## 五、组织结构审查

### 清晰分段

| 检查项   | 标准                 | 问题           |
| :------- | :------------------- | :------------- |
| 章节划分 | 每个章节单一主题     | 主题混杂需拆分 |
| 标题层级 | 最多 3 层（##、###） | 过深需扁平化   |
| 段落长度 | 每段 < 150 词        | 过长需拆分     |
| 列表项   | 每项 < 30 词         | 过长需独立成段 |

## 七、审计流程

### Phase 1: 快速扫描（10 分钟）

1. 检查文件总行数
2. 识别重复章节标题
3. 标记过长段落和列表

### Phase 2: 内容审查（30 分钟）

1. **单一事实来源**：标记重复内容
2. **抽象高度**：标记显而易见内容
3. **示例优于规则**：标记可转为示例的规则
4. **防御性重复**：标记过度强调

### Phase 3: 优化实施（1 小时）

1. 删除显而易见内容
2. 合并重复内容到单一位置
3. 转换规则为示例
4. 精简过度强调

### Phase 4: 验证（10 分钟）

1. 检查是否仍有重复
2. 确认抽象高度合适
3. 验证示例清晰度
