# NestJS Better Auth 集成后续工作

从 Phase 1 实现中延期的想法与增强项。

**当前状态：** Phase 1 已完成 ✅ (2024-03-02)  
**测试覆盖率：** 67.5%（86 个测试，100% 通过）  
**下一阶段：** Phase 2 - 增强功能（预计 1-2 周）

## 增强项

### 1. 测试覆盖率提升（P1）

**当前状态：** 67.5% 覆盖率
**目标状态：** 80%+ 覆盖率

**需要补充的测试：**
- AuthModule 中间件配置（需要 E2E 测试）
- 钩子系统（需要 E2E 测试）
- 错误处理边界情况
- GraphQL/WebSocket 集成测试

**预计工作量：** 3-5 天

### 2. E2E 测试示例项目（P1）

**当前状态：** 缺少完整的 E2E 测试
**增强方案：**
- 创建示例项目展示完整认证流程
- 包含 HTTP、GraphQL、WebSocket 示例
- 包含角色权限示例
- 包含钩子系统示例

**预计工作量：** 2-3 天

### 3. Fastify 适配器支持（P2）

**当前状态：** 仅支持 Express 适配器
**增强方案：**
- 创建 Fastify 适配器抽象层
- 修改中间件以支持 Fastify
- 更新配置选项
- 添加 Fastify 测试和示例

**优先级：** P2
**预计工作量：** 5-7 天

**参考：** Fastify 的 request/response 对象与 Express 不同

### 4. 组织角色查询缓存（P2）

**当前状态：** 每次请求都查询组织角色
**增强方案：**
- 使用 LRU Cache 缓存组织角色
- 缓存时间 5 分钟
- 支持手动清除缓存
- 添加缓存命中率监控

**优先级：** P2
**预计工作量：** 1-2 天

**收益：** 减少数据库查询，提升性能

### 5. 装饰器组合（P3）

**当前状态：** 需要手动组合多个装饰器
**增强方案：**
- 提供 `@AdminOnly()` = `@Roles(['admin'])`
- 提供 `@OrgOwnerOnly()` = `@OrgRoles(['owner'])`
- 支持自定义装饰器组合

**优先级：** P3
**预计工作量：** 1 天

**收益：** 减少样板代码，提升开发体验

### 6. 函数式 trustedOrigins 支持（P3）

**当前状态：** 不支持函数式 trustedOrigins
**增强方案：**
- 支持函数形式的 trustedOrigins
- 动态计算允许的 origins
- 保持向后兼容

**优先级：** P3
**预计工作量：** 2-3 天

## 技术债

### 1. AuthModule 覆盖率较低（39.72%）

**当前状态：** 中间件和钩子配置代码缺少测试
**改进方案：**
- 添加 E2E 测试覆盖完整流程
- 使用 supertest 测试中间件
- 使用 NestJS 测试工具测试生命周期

**优先级：** P1
**预计工作量：** 2-3 天

### 2. 错误处理不够统一

**当前状态：** 不同上下文错误处理方式不一致
**改进方案：**
- 定义统一的错误码
- 使用 NestJS 异常过滤器
- 统一错误响应格式

**优先级：** P2
**预计工作量：** 1-2 天

### 3. 日志记录不完整

**当前状态：** 关键操作缺少日志
**改进方案：**
- 记录认证成功/失败事件
- 记录角色检查失败事件
- 使用结构化日志（JSON）
- 支持日志级别配置

**优先级：** P2
**预计工作量：** 1 天

## 性能优化建议

### 短期（Phase 2）

1. **组织角色缓存**：使用 LRU Cache，减少 80% 数据库查询
2. **元数据缓存**：缓存 Reflector 结果，减少反射开销
3. **会话提取优化**：避免重复的 session 查询

### 长期（Phase 3+）

1. **分布式缓存**：使用 Redis 缓存组织角色
2. **连接池优化**：优化数据库连接池配置
3. **异步处理**：将审计日志异步化

## 优先级说明

- **P0 (Critical)**：核心功能，必须实现
- **P1 (High)**：重要功能，优先实现
- **P2 (Medium)**：有价值，可延后实现
- **P3 (Low)**：可选功能，根据需求实现
